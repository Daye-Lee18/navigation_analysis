---

## 1. 과제 문제 해석

### 1-1. 불만 내용 (VOC)

> 평소에는 목적지까지 **10km 거리**로 주행하였으나,  
> 내비게이션이 **30km 우회 경로**를 제공하여 이상 경로 불만 접수됨.

### 1-2. 데이터 문제로 변환

| VOC 표현 | 데이터 문제로 변환 |
|----------|------------------|
| 평소 10km 경로 | 최단/최소시간 경로(Route A) — 내비가 평소 추천하던 경로 |
| 30km 우회 경로 | 내비가 새로 선택한 Route B — 거리 3배, 통행시간도 더 김 |
| "갑자기" 우회 | 경로 추천 cost function의 입력값(속도, 제한정보)이 변경됨 |

**핵심 질문:**
> 내비게이션의 cost function이 왜 더 짧고 빠른 Route A 대신  
> Route B(우회)를 선택했는가?

## 3. 가설 설정 (Hypothesis)

| # | 가설 | 내용 |
|---|------|------|
| H1 | traffic 과대 추정 | Route A 속도를 실제보다 낮게 추정 → cost 역전 → Route B 선택 |
| H2 | road restriction 오류 | 해제된 도로 통제 정보가 내비 DB에 남아 Route A 회피 |
| H3 | cost function bias | 거리 vs 시간 가중치 오설정 → 더 긴 경로가 낮은 cost |
| H4 | rerouting 로직 문제 | 재탐색 임계값이 너무 낮아 일시적 혼잡에도 불필요한 재탐색 |

## 5. 가설 검증

### H1. Traffic 과대 추정

**분석:** Route A 속도가 얼마나 낮아져야 Route B가 선택되는지 임계값 계산

**Cost function 분기점 (8시 기준):**

```
Route A 통행시간 = Route B 통행시간 조건:
  dist_A / speed_A_est × 60 = 34분 (Route B 8시 통행시간)
  18.5km / speed_A_est × 60 = 34분
  → speed_A_est = 32.5 km/h
```

| 조건 | Route A 추정 속도 | Route A 예측 시간 | 선택 경로 |
|------|-----------------|-----------------|---------|
| 정상 (실제) | 48.3 km/h | 23분 | **Route A** |
| 10% 과소추정 | 43.5 km/h | 26분 | **Route A** |
| 30% 과소추정 | 33.8 km/h | 33분 | **Route A** |
| **33% 과소추정** | **32.5 km/h** | **34분** | **→ Route B 선택** |
| 50% 과소추정 | 24.2 km/h | 46분 | Route B |

**결론:** Route A 속도를 **실제 대비 33% 이상 과소추정**해야 우회 경로 선택됨

**날짜별 실제 속도 분석 (3월 31일, 8시):**

| 지표 | Route A (올림픽대로) | Route B (동일로·천호대로) |
|------|---------------------|------------------------|
| 평균 | 48.3 km/h | 32.2 km/h |
| 표준편차 | **24.2 km/h** | 0.9 km/h |
| 최저 | 19.9 km/h | 30.6 km/h |
| 최고 | 84.8 km/h | 33.8 km/h |

- Route A: **편차가 매우 큼** (cv ≈ 0.50) → 날에 따라 극단적으로 달라짐
- Route B: **거의 일정** (cv ≈ 0.03) → 도시고속도로보다 훨씬 안정적

**31일 중 10일(32%)은 Route A 속도가 임계값(32.5 km/h) 이하로 실제로 느렸음**

→ **이 10일에는 우회 경로 선택이 정당화될 수 있음**  
→ 나머지 21일(68%)에 우회가 발생했다면 → **과대 추정 오류 또는 H2 원인**

> 📌 **시각화:** `output/voc2_h1_cost_function.png`, `voc2_h1_daily_speed.png`

### H2. Road Restriction 오류

**가설:** 올림픽대로 구간에 발생했던 사고·공사·통제가 해소됐음에도  
내비게이션 DB에 제한 정보가 남아 있어 Route A를 지속적으로 회피

**확인 포인트:**

| 확인 항목 | 방법 | 기대 결과 |
|----------|------|----------|
| 올림픽대로 링크의 AccInfo 이벤트 | `link_id` 매핑 | 해소된 이벤트 존재 여부 |
| `exp_clr_date/time` < 현재 시각 | API 쿼리 | 해소됐는데 남아있는 이벤트 |
| 내비 DB 갱신 주기 | 내부 시스템 확인 | 갱신 lag (분/시간 단위) |

**분석 시나리오:**
```
[사고 발생] 올림픽대로 구간 통제 → AccInfo 등록
  ↓
[사고 해소] exp_clr_date 도래 → 실제 도로 통행 가능
  ↓
[내비 미갱신] 내비 DB 갱신 lag 존재 → 여전히 Route A 회피
  ↓
[결과] 사용자: "왜 멀쩡한 올림픽대로를 피해가지?"
```

**결론:** 도로 통제 해소 후 **내비 DB에 반영되는 갱신 주기(lag)**가 VOC2의  
직접적 원인일 가능성이 높음 — 특히 10일이 아닌 다른 날 우회가 발생했다면

> 📌 **결과:** `output/voc2_h2_road_restriction.json`

### H3. Cost Function Bias

**가설:** 내비게이션 경로 탐색 cost function에서  
거리(distance) 가중치 β가 너무 낮아 긴 경로의 패널티가 불충분

**Cost Function 구조:**
```
cost(경로) = α × 시간(분) + β × 거리(km) + γ × 기타
```

**분기점 분석 (Route A vs Route B, α=1 고정):**
```
cost_A = cost_B 조건:
  23 + β × 18.5 = 34 + β × 18.3
  β × (18.5 - 18.3) = 34 - 23
  β × 0.2 = 11
  β = 55.52 분/km
```

**해석:**
- 두 경로의 거리 차이가 0.2km로 매우 작아 **β가 극단적으로 커야 cost 역전** 발생
- 이 시나리오에서는 거리 가중치보다 **속도 추정 오류(H1)나 통제 정보(H2)**가 더 직접적 원인
- 단, 원래 VOC의 10km vs 30km 시나리오라면:
  ```
  β × (30 - 10) = 시간 차이 → β가 상대적으로 작아도 cost 역전 가능
  → 거리 격차가 클수록 H3(cost function bias)의 영향이 커짐
  ```

**결론:** 거리 차이가 클수록 H3의 영향력 증대. 10km vs 30km 원래 시나리오에서는  
β 설정 오류가 주요 원인일 수 있음.

> 📌 **시각화:** `output/voc2_h3_cost_sensitivity.png`

### H4. Rerouting 로직 문제

**가설:** 재탐색 트리거 임계값이 낮게 설정되어  
Route A의 일시적 속도 저하 시 불필요하게 재탐색 → 더 긴 Route B 안내

**Route A 시간대별 속도 변동 분석:**

| 시간 | 평균 속도 | 표준편차 | 임계(32.5km/h) 미달 위험 |
|------|---------|---------|----------------------|
| 7시 | 63.3 km/h | - | 낮음 |
| 8시 | 48.3 km/h | ±24.2 | **높음 (31일 중 10일)** |
| 9시 | 47.5 km/h | - | 높음 |

**문제 구조:**
```
[정상 상태] Route A 속도 48 km/h → Route A 선택
  ↓ 일시적 혼잡 (교통 사고, 점심 러시 등)
[임계 이하] 속도 30 km/h → 재탐색 트리거
  ↓ 재탐색 → Route B 선택
[혼잡 해소] Route A 속도 회복 → 하지만 이미 Route B로 안내 중
  ↓
[결과] 사용자: "멀쩡해진 도로 놔두고 왜 돌아가?"
```

**임계값 민감도:**
- 임계값이 낮으면: 자주 재탐색 → 불필요한 우회 빈발
- 임계값이 높으면: 재탐색 안 함 → 심각한 혼잡도 방치

**결론:** 재탐색 트리거를 단순 속도 임계값이 아닌  
**지속 시간 조건 추가**(예: 3분 이상 지속될 때만 재탐색)가 필요

> 📌 **시각화:** `output/voc2_h4_rerouting.png`

### 6-1. 가설 검증 결과

| 가설 | 결론 | 근거 |

| H2. road restriction 오류 | ✅ **가장 유력** | 해소된 통제 정보 미갱신이 지속적 우회의 직접 원인 |

| H4. rerouting 로직 | ✅ **유력** | 일시적 혼잡 → 재탐색 → 우회 고착 패턴 설명 |

### 6-2. Root Cause 정리

```
[VOC2 이상 경로 Root Cause]

직접 원인:
  1. 도로 통제 정보 갱신 lag (H2) — 해소된 통제가 내비에 남아 Route A 회피
  2. 재탐색 임계값 과민 (H4)   — 일시적 혼잡에 재탐색 → Route B 고착

간접 원인:
  3. Traffic 추정 오류 (H1)    — Route A 속도 과소추정 시 cost 역전
  4. Cost function β 과소 (H3) — 거리 패널티 부족으로 긴 경로 불이익 미흡

결과: 정상 도로를 회피하거나 일시적 혼잡에 과민 반응해 3배 우회
```

## 7. AI 개선안 및 미래 방향

### 7-1. 단기 개선

| 항목 | 내용 |
|------|------|
